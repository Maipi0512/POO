Los 4 pilares de POO aplicados

1 — Encapsulamiento

Todos los atributos son private. Ninguna clase externa puede leerlos o modificarlos directamente; solo a través de getters y setters.

// En Proveedor.java
private double topeMaximoDeuda;           // nadie lo toca desde afuera directamente

public void setTopeMaximoDeuda(double v) { topeMaximoDeuda = v; }
public double getTopeMaximoDeuda()       { return topeMaximoDeuda; }

¿Por qué? Si el atributo fuera public, cualquier código podría poner un valor negativo y romper la lógica que evita el exceso de deuda.

2 — Herencia

Tres jerarquías eliminan código duplicado. La clase padre define los atributos y métodos comunes; los hijos solo agregan o redefinen lo que los diferencia.

Comprobante  (abstract)      numero, fechaEmision, saldoPendiente, registrarPago()...
  Factura                    agrega lista de OC asociadas
  NotaCredito                igual que Comprobante, solo cambia afectarCuentaCorriente()
  NotaDebito                 igual que Comprobante, solo cambia afectarCuentaCorriente()

Impuesto  (abstract)         porcentajeBase, minimoNoImponible, escalas...
  ImpuestoIVA                solo para Responsables Inscriptos
  ImpuestoIngresosBrutos     para todos
  ImpuestoGanancias          excluye Monotributistas

MedioPago  (abstract)        idMedioPago, importe, fecha
  Efectivo
  TransferenciaBancaria
  Cheque  (abstract)
    ChequePropio
    ChequeTerceros

3 — Polimorfismo

El mismo mensaje se envía a objetos distintos y cada uno responde según su tipo real, sin necesidad de if/else.

DóndeLlamada polimórficaQuién responde / resultado
Proveedor.obtenerCuentaCorriente()c.afectarCuentaCorriente()Factura: +saldo · NotaCredito: -saldo · NotaDebito: +saldo
SistemaGestionCompras.calcularRetenciones()imp.calcularRetencion(base, prov)IVA · IIBB · Ganancias tienen reglas distintas
OrdenPago.procesarPago()mp.procesarPago()Efectivo: true siempre · Transferencia: valida CBU · Cheque: valida campos y fechas

4 — Abstracción

Las clases abstractas definen qué debe hacer algo pero no cómo. No se pueden instanciar directamente.

new Comprobante(...)    // ERROR: es abstracta
new Factura(...)        // OK: clase concreta

new Impuesto(...)       // ERROR: es abstracta
new ImpuestoIVA(...)    // OK: clase concreta

Esto garantiza que siempre se use una clase con comportamiento real y no una incompleta.

Enums — farmared.enums (8 archivos)

Los enums garantizan que solo se usen valores válidos. Es imposible asignar un estado inventado:

oc.setEstado("aprobada");                    // Error de compilación
oc.setEstado(EstadoOrdenCompra.AUTORIZADA);  // OK

CondicionIVA.java

Clasificación tributaria del proveedor. Usada en calcularRetencion() para decidir qué impuestos aplican.

RESPONSABLE_INSCRIPTO  le retienen IVA + IIBB + Ganancias
MONOTRIBUTISTA         le retienen IVA + IIBB (no Ganancias)
EXENTO                 le retienen solo IIBB
CONSUMIDOR_FINAL       no es proveedor habitual

TipoIVA.java

Alícuota de IVA de un producto. Cada valor lleva su número para usarlo directamente en cálculos.

IVA_21   getPorcentaje() devuelve 21.0
IVA_10_5 getPorcentaje() devuelve 10.5
EXENTO   getPorcentaje() devuelve 0.0

TipoImpuesto.java

Identifica el tipo de retención: IVA, INGRESOS_BRUTOS, GANANCIAS. Usado en certificados de no retención y en reportes agrupados por tipo.

TipoComprobante.java

FACTURA_A/B/C, NOTA_CREDITO, NOTA_DEBITO. Determina el tipo de comprobante fiscal.

EstadoOrdenCompra.java

EMITIDA → AUTORIZADA → PARCIALMENTE_FACTURADA → FACTURADA → ANULADA

No se puede anular una OC en estado FACTURADA (RF-13).

EstadoOrdenPago.java

GENERADA → EMITIDA → ANULADA

EstadoComprobante.java

PENDIENTE → PAGADO_PARCIAL → PAGADO_TOTAL → ANULADO

registrarPago(monto) en Comprobante avanza el estado automáticamente.

RolUsuario.java

OPERADOR      carga datos, emite OC y OP
SUPERVISOR    puede autorizar excepciones (tope de deuda, desvíos de precio)
ADMINISTRADOR gestión de usuarios y parametrización

public PrecioAcordado obtenerUltimoPrecio(Proveedor proveedor) {
    PrecioAcordado resultado = null;
    for (PrecioAcordado pa : preciosAcordados) {
        if (pa.getProveedor().equals(proveedor) && pa.estaVigente()) {
            if (resultado == null ||

Módulo 3 — Impuestos y Retenciones (m3_impuestos)

EscalaRetencion.java — RF-10

Define un tramo de monto con su porcentaje. Ejemplo: de $0 a $10.000 aplica el 3%; de $10.001 en adelante aplica el 5%.

private double montoDesde;
private double montoHasta;   // 0 = sin límite superior

public boolean aplicaA(double monto) {
    boolean superaMinimo = monto >= montoDesde;
    boolean bajoMaximo   = (montoHasta == 0) || (monto <= montoHasta);
    return superaMinimo && bajoMaximo;
}

Impuesto.java (abstracta) — RF-08

Clase base para los tres impuestos. Define los atributos comunes y el método abstracto calcularRetencion().

protected double porcentajeBase;        // porcentaje plano si no hay escalas
protected double minimoNoImponible;     // mínimo debajo del cual no se retiene
protected List<EscalaRetencion> escalas;

// Busca el porcentaje según escalas; si no hay escalas, usa porcentajeBase
public double determinarPorcentaje(double base) {
    for (EscalaRetencion e : escalas) {
        if (e.aplicaA(base)) return e.getPorcentaje();
    }
    return porcentajeBase;
}

// Cada subclase implementa su propia lógica
public abstract double calcularRetencion(double base, Proveedor prov);

ImpuestoIVA.java

public double calcularRetencion(double base, Proveedor prov) {
    if (prov.getCondicionIVA() != CondicionIVA.RESPONSABLE_INSCRIPTO) return 0.0;
    if (base <= minimoNoImponible) return 0.0;
    return base * (determinarPorcentaje(base) / 100.0);
}

Solo retiene a Responsables Inscriptos. Monotributistas y Exentos devuelven 0.

ImpuestoIngresosBrutos.java

public double calcularRetencion(double base, Proveedor prov) {
    if (base <= minimoNoImponible) return 0.0;
    return base * (determinarPorcentaje(base) / 100.0);
}

Aplica a todos los proveedores, sin distinción de condición IVA.

ImpuestoGanancias.java

public double calcularRetencion(double base, Proveedor prov) {
    if (prov.getCondicionIVA() == CondicionIVA.MONOTRIBUTISTA) return 0.0;
    if (base <= minimoNoImponible) return 0.0;
    return base * (determinarPorcentaje(base) / 100.0);
}

Exime a Monotributistas. Todos los demás tributan.

procesarPago()

public boolean procesarPago(List<MedioPago> lista) {
    double total = 0.0;
    for (MedioPago mp : lista) {
        if (!mp.procesarPago()) return false;   // polimorfismo: cada medio se valida solo
        mediosPago.add(mp);
        total += mp.getImporte();
    }
    // La suma de medios debe coincidir con el neto (tolerancia 1 centavo)
    return Math.abs(Math.round(total * 100.0) / 100.0 - importeNeto) < 0.01;
}

public void cerrarOrden() {

}

calcularRetenciones() — método privado

private void calcularRetenciones(OrdenPago op, Proveedor prov, Date fecha) {
    double base = op.getImporteBruto();
    for (Impuesto imp : impuestos) {
        if (prov.tieneCertificadoVigente(imp.getTipo())) continue;  // RF-09: salta si tiene cert
        double importe = imp.calcularRetencion(base, prov);          // polimorfismo
        if (importe > 0)
            op.agregarRetencion(new Retencion(imp, base,
                imp.determinarPorcentaje(base), importe));
    }
}

¿Por qué new Date(125, 0, 1)?

Es la forma antigua de crear una fecha. El primer parámetro es el año restado 1900, por eso 125 = 2025. El compilador da una advertencia (deprecated) pero funciona correctamente. La forma moderna requeriría usar LocalDate, que implica más cambios al código.
